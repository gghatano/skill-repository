# Adapter: Generic（サブエージェント無し環境）

サブエージェント/並列実行機構を持たない Coding Agent でも Skill/Workflow を実行するための
対応づけ。**構造は同じまま、並列を逐次隔離で代替**する。

## 抽象操作 → Generic

| 抽象操作 | Generic での実行 |
| --- | --- |
| 独立サブエージェント N 体 | **逐次**に N 回実行する（1 回 = 1 レビュアー/1 候補） |
| Context 隔離 | 各回の前に Context をリセットし、共有物（`spec.md`・対象）だけを与える |
| バリア | N 回分の成果物ファイルが揃ってから統合ステップへ進む |
| 並行実装の衝突回避 | ブランチ/ディレクトリを分けて 1 タスクずつ |
| モデルティア | 単一モデルでも可（役割ごとにプロンプトで分離） |
| 成果物受け渡し | リポジトリ上のファイル（この Adapter でも共通） |

## 独立性を保つ逐次実行

```
for r in [correctness, security, performance]:
    Context をリセット
    入力 = spec.md + 対象（他レビュー結果は渡さない）
    review を実行 → review-<r>.md
統合: review-*.md を集めて integrate → review.md
```

ポイントは、逐次でも **各回に他の回の出力を見せない**こと。これで「同じ前提への
早期収束」を防ぐという独立探索の目的が達成できる（速度は並列に劣る）。

## トレードオフ

- 速度: 並列 Adapter より遅い（N 倍の逐次時間）。
- 品質: 隔離を守れば独立性は同等。
- コスト: 追加 Agent プロセスが不要な分、単純。

## 使いどころ

- 単一エージェントしか使えない CI 環境や軽量ランタイム。
- まず Generic で構造を回し、並列 Adapter へ移すと段階的に高速化できる（成果物駆動なので移行が容易）。
