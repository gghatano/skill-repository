# 実験 Issue 作成スキル

## 目的

新しい合成データ比較実験のアイデアを、既存の実験 Issue (#4 exp02 / #5 exp03 / #6 exp04) と同じ構造で GitHub に登録する。情報が足りない部分は **ユーザに質問してから** Issue を作る。

**鉄則**: 不十分な情報で Issue を作らない。欠落があれば必ず `AskUserQuestion` で聞き返す。

## 実行フロー

### Step 1. 入力を解釈する

ユーザの発話から以下を抽出する:

| フィールド | 例 |
|---|---|
| 実験 ID (exp0N) | `exp05` — 既存 #4/#5/#6 の次 |
| 実験タイトル | `exp05: PrivBayes の次数感度実験` |
| 比較軸 | アルゴリズム / 条件パラメータ / データセット / 前処理 / ライブラリ |
| 目的 (何を明らかにしたいか) | 1 行で言えるか |

ユーザが曖昧なら Step 2 の質問で補う。

### Step 2. 欠落フィールドを質問する

以下 6 フィールドのうち発話から埋まっていないものを `AskUserQuestion` で聞く。**1 回の質問呼び出しに複数項目をまとめてよい**。

必須フィールド:

1. **背景 / 目的** — なぜこの実験をやるのか、何を明らかにしたいのか (1〜3 行)
2. **スコープ (具体タスク)** — チェックリスト形式で追試可能な粒度に。例:
   - どのアルゴリズム/実装を追加するか
   - どの条件グリッド (epsilon / seed / n)
   - どのデータセット
   - 何を成果物 (code / config / report) として作るか
3. **評価指標 / 見所** — 既存の指標で足りるか、追加が必要か。特に重要な指標は何か
4. **見積もり** — 計算コスト (fit 時間 × runs)、新規実装コスト (新しい wrapper の要否)
5. **前提・依存** — 既存共通基盤 (shared/) をどこまで流用できるか、他 Issue への依存
6. **優先度の目安** — 高 / 中 / 後回し可 と、その理由 1 行

質問は冗長にしない。ユーザの発話に既にある情報は**聞き直さない**。例えばユーザが「exp05 で Bank データセットに MST を適用」と言っていたら、対象とデータセットは埋まっているので背景・スコープ・見積もりだけ聞く。

### Step 3. Issue 本文を組み立てる

以下のテンプレを埋める。**既存 Issue #4〜#6 のスタイルに合わせる** (チェックボックス、優先度コメント、完了条件)。

```markdown
## 背景

{{背景 / 目的を 1〜3 行。既存実験の結論をどう前進させるか}}

## スコープ

- [ ] {{具体タスク 1}}
- [ ] {{具体タスク 2}}
- [ ] `docs/report_{{NN}}_{{shortname}}.md` 作成 (テンプレ: `docs/templates/report_template.md`)
- [ ] `results/archive/<date>_exp{{NN}}/` に前実験を退避してから実行

## 評価指標 / 見所

- {{重要指標とその読み方}}
- {{既存指標で足りない部分があれば明記}}

## 見積もり

- 計算コスト: {{fit 時間 × runs}}
- 実装コスト: {{新規 wrapper/runner の要否、既存共通基盤で済むか}}

## 前提・依存

- {{既存 shared/ 基盤のうち流用する部分}}
- {{他 Issue との依存関係、あれば #N で参照}}

## 完了条件

- [ ] {{主要指標の比較表と代表図が出る}}
- [ ] レポートが `docs/templates/report_template.md` に沿っている
- [ ] 7 観点レビューエージェント (`docs/review.md`) を通している
- [ ] README §1 の実験一覧に行を追加した

## 優先度の目安

**{{高 / 中 / 後回し可}}**: {{理由 1 行}}
```

### Step 4. Issue を作成する

```bash
gh issue create \
  --title "exp{{NN}}: {{短いタイトル}}" \
  --body-file <(cat <<'EOF'
{{組み立てた本文}}
EOF
) \
  --label "experiment{{,backlogなら ,backlog}}"
```

labels:
- **experiment** は必須
- 優先度「後回し可」なら **backlog** も付ける
- 高優先度なら experiment のみ

### Step 5. ユーザに報告する

以下を 1 メッセージで返す:

- 作成した Issue 番号と URL
- タイトル
- 優先度の目安 (高 / 中 / 後回し可)
- README §1 の「候補実験」表に追加する行のスニペット (ユーザが別 PR で反映できるように)

## QA 時の注意

- 「背景」が 1 行しか無くても、「何を言えたら成功か」を追加で聞くと Issue の解像度が上がる
- ユーザが「何となく試したい」レベルの場合は **goal 駆動** で質問する:
  - 「この実験で何が言えたら成功ですか?」
  - 「既存の実験 01 の結論とどう違う結論を期待しますか?」
- 計算コスト見積もりが立たない場合は、まず 1 run smoke test で実測する提案を Issue のスコープに入れる

## 既存 Issue の参考

- Issue #4 (exp02): epsilon 両端 + フル Adult — backlog 例
- Issue #5 (exp03): PrivBayes / AIM 追加 — 高優先度の例
- Issue #6 (exp04): 別データセット横展開 — 中優先度の例

これらの形式・粒度を揃えるのが本スキルの目的。

## 実行例

**ユーザ**: 「MST に top-K の有無を変えた感度実験を issue にして」

**スキルの動き**:
1. 抽出: 対象=MST、軸=前処理の top_k 有無、データ=Adult (デフォルト)
2. `AskUserQuestion` で欠落 4 項目を質問:
   - 背景: なぜこの実験? (「`native-country` 圧縮の影響を isolate するため」など)
   - スコープの詳細: top_k=None / top_k=10 / top_k=20 の 3 条件?
   - 見積もり: 計算コストの概算
   - 優先度: 今すぐ? それとも backlog?
3. 回答を反映して本文組み立て
4. `gh issue create --label experiment,backlog`
5. ユーザに URL と README スニペットを返す
