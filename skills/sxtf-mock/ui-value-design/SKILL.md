# ui-value-design — 価値で語るUI（開発・レビュー）

sxtf-mock の画面を **「何ができるか(機能)」ではなく「使う人に何が嬉しいか(価値)」** で語るためのSKILL。契約・ドメインは `.claude/knowledge/`、限界は `docs/利用ガイド_検証できること.md` に従う（UIは契約・スコープ・限界と矛盾させない）。

## 中核原則（value over feature）

1. **Job-to-be-done で見出す**：見出しはユーザーの問い／得たい結果で書く。機能名・No.X を主役にしない。
   - ✗「来歴・経路検証（No.9）」 → ✓「受け取った材料が“確かな出どころ”か確かめる」
   - ✗「真正性チェック（No.2/3）」 → ✓「この証明書は本物で、正しく届いたか」
2. **結論(verdict)＋“だから何”**：判定結果は結論を大きく、その意味（安心して使える／使用前に要確認）を添える。
3. **専門用語を平場へ翻訳**（技術語は括弧で添える程度に降格）。下表参照。
4. **段階的開示**：価値・結論を上に、仕組み・詳細（経路DAG・ハッシュ等）は折りたたみ/オンデマンドで下に。
5. **誠実さ**：✓を過信させない。PoCの限界（署名=なりすまし耐性なし／台帳=ブラウザ内／非開示≠暗号秘匿／現物はスコープ外）と矛盾しない。誇大広告にしない。
6. **機能Noは降格**：消さず、デモ評価者向けに小さく（フッタ/ツールチップ/「内部対応: No.9」程度）。画面の主役にしない。

## 専門用語 → 平場の日本語（統一）

| 技術語 | UI表記 |
|--------|--------|
| verdict | 結論／判定（「正しく流通した本物」） |
| content | 証明書の内容（中身） |
| route / route detail | 経路・来歴／経路の詳細 |
| provenance | 来歴 |
| DAG | （表に出さない。「分割・併合を含むつながり」） |
| masked / 非開示 | 「あなたには開示されていません」（－。NGではない旨を添える） |
| owner | 発行者 |
| operator / auditor / admin | 事業者／検査機関(監査)／管理者 |
| transfer / ledger | 授受／授受の記録 |

## ペルソナ別の提供価値（見出し・導入の素材）

- **事業者(operator)**：受け取った材料を**安心して使える**こと／自分が出した証明に**責任と証跡が残る**こと／**知らない素材・知らない相手**が紛れていないと分かること。
- **検査機関(auditor)**：**中身を見せてもらわなくても**、正しい経路で本物の起点に遡れると**確認できる**こと（No.9×No.4）。
- **管理者(admin)**：誰が使えて何を見られるかを**管理できる**こと。

## コピーの型（各画面）

- **画面タイトル**：価値（ユーザーの問い/結果）。例「受け取った証明書の信頼性を確認」
- **サブ見出し**：その画面でできること＋得られる安心を1〜2文。
- **結論ブロック**：✓/✗ を大きく＋一言の意味。根拠は箇条書きで下に。
- **セクション名**：「証明書の内容」「どこから来たか（来歴）」等の平場表現。
- **空状態**：価値志向（「確かめたい書類を選んでください」）。
- **限界の注記**：必要箇所に控えめに（利用ガイドへの導線）。

## ナビゲーション（タブ）の扱い

タブは**道案内**なので見出しと別ルール：**短く・予測可能**にする（価値の文にしない＝長く・似た響きで探しにくくなる）。ただし**システムの仕様語でなくユーザーのタスク語**で書く。本プロジェクトの確定ラベル：

| href | ラベル（確定） |
|------|----------------|
| `/` | ホーム |
| `/sender` | 発行・送信 |
| `/inbox` | 受け取り |
| `/provenance` | 信頼性チェック |
| `/access` | 開示設定 |
| `/audit` | 履歴・進捗 |
| `/admin` | 利用者・権限管理 |

featureNo はナビ本体に出さない（降格。ランディングのカード等で小さく添える程度）。

## 視覚・一貫性

- 3状態は **✓green / ✗red / －gray（非開示・NGではない）** で統一（既存規約）。
- shadcn/最小primitive を流用、余白・階層を揃える。色は意味（OK/NG/非開示/注意）。
- 日本語UI。用語は glossary＋上表に統一。
- アクセシビリティ最低限：十分なコントラスト、ボタン/フォームにラベル、見出し階層。

## レビュー・チェックリスト（画面ごとに self-review）

- [ ] 見出しは**価値/結果**で書かれているか（機能名・No.X が主役になっていないか）
- [ ] ユーザーの「自分は何が嬉しいか」に最初の画面で答えているか
- [ ] 結論(verdict)が主役で、“だから何”が添えてあるか
- [ ] 専門用語が平場に翻訳されているか（verdict/content/route/DAG 等）
- [ ] 仕組み・詳細は段階的開示（上=価値、下=詳細）になっているか
- [ ] PoCの限界・スコープと矛盾していないか（誇大広告でないか）
- [ ] 機能No は降格（小さく）されているか
- [ ] 3状態・用語・余白が他画面と一貫しているか
- [ ] アクセシビリティ最低限（コントラスト/ラベル/見出し）
- [ ] 契約・データ操作は変更していない（UIコピー/IAのみ。型・store・provenance 不変）

## 進め方（develop / review）

1. 対象画面を `preview_start`＋`preview_screenshot` で現状把握。
2. 上記の型・チェックリストで見出し/コピー/IAを value-first に書き換え（**契約ファイルは触らない**＝UIのみ）。
3. `preview_screenshot` で before/after を確認。1〜2画面で型を固め、合意後に全画面へ展開。
4. ブランチ `feat/ui-*` で作業し、DoD（build＋既存E2E green）を満たして develop へ。
