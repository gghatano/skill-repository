# 実験計画（公平な比較と業務シナリオの設計）

本プロセスは判断が中心のため、サブエージェントに委譲する場合は
`planner`（上位ティア、`.claude/agents/planner.md`）を使う。

`report-skeleton` が出力した「必要な実験一覧」を入力に、実行可能な計画書を作る。
計画は `docs/plans/<ID>-<slug>.md` に置き、実行（`experiment-run`）と結果整理（`report-review`）から参照する。

## 0. 論文準拠プロトコルの抽出（再現追試では最優先）

タスクが**論文の再現追試**である場合、実験を設計する前に**論文の実験セクションを一次情報として読み**、
次を抽出して計画書冒頭の「論文準拠プロトコル」表に固定する。**ここを飛ばして手近なコードを動かすと、
論文ではなく「コードの example」を追試してしまう**（前処理・指標・データ規模が論文と食い違う事故が実在する）。

- **データセット**: 正確な規模（行数・列数等）と由来。論文が使った前処理（欠損の扱い・除外・スケーリング・外れ値）。
- **主要評価指標**: 定義式まで。論文が予測誤差（例: 相対誤差）なら別の代理指標で代用しない（独自指標は併記に留める）。
- **比較手法・ベースライン**、**主要パラメータ**（手法のハイパーパラメータ・実験条件）。
- **照合対象**: 論文の図表番号と報告値（後で自分の結果を並べて突き合わせるため）。

**一次情報の優先順位（厳守）**: **論文 ＞ 公式リポジトリ/コード ＞ example・tutorial**。
三者が食い違うとき「何を再現するか」は論文が正。論文から逸脱する設計（前処理の改善・指標の変更を含む）は、
後述の「逸脱ログ」に**理由つきで明示**し、レポート本文でも断る（黙って改善しない）。
一次情報の読解そのものは `related-info-review` / `planner` に委譲してよい。

**再現ターゲットの宣言（必須）**: 実験計画の冒頭で `exact`（数値完全一致を狙う）/ `trend`（傾向の再現を狙う）を
**明示**し、理由（RNG 非再現性・縮小規模・実装差など。規約 §4「乱数再現性と再現ターゲット」参照）を添える。
非決定的な処理（非 seedable な乱数・外部サービス・並列スケジューリング依存）を含む実験は原則 `trend`。
さらに**論文本文と公開コードの食い違い**（実験条件・データ規模・試行数など）がある場合は、計画書に
差分表（行=項目、列=論文／公開コード既定／本追試の採用値＋理由）として開示する。曖昧なまま「再現」と称さない。

## 0.5 多数モデル学習を伴う実験のコストモデル

**N 個の独立な学習・試行が支配的**な実験（多数モデルの fit・大規模スイープ等）は、着手前にコストを宣言する。

1. **コスト式の明記**: 総時間 ≈ `N_runs × per_run_sec / effective_parallelism`。
   `N_runs`（= 試行数 × 条件数等）と `per_run_sec` を計画書に書く。
2. **事前スモーク → 外挿（必須）**: 本実行前に小規模（例 N を 1/50）で per-run を実測し、総時間を**宣言**してから着手する。
   この「スモーク→外挿」は長時間実験の共通定石で、`experiment-run` の進捗・可観測性節も同じ手順を参照する。
3. **並列効率の現実性チェック**: 理論並列数どおりに速くならない要因（I/O・entropy の直列化・メモリ帯域・
   BLAS オーバーサブスクリプション。詳細は `experiment-run` の並列・堅牢性節）を想定し、ETA が乖離したら
   **worker 数を増やす前にボトルネックを疑う**。
4. **縮小規模の意思決定**: コストが許容を超える場合の縮小（N_runs を減らす／収束図で補う）と、その統計的代償
   （解像度低下）を計画に明記し、完走 vs 縮小をユーザーと合意する。

## 計画書テンプレート

```markdown
# 実験<ID>: <タイトル>
- 再現ターゲット（再現追試の場合）: exact / trend ＋理由（RNG 非再現性・縮小規模・実装差）。
  上流コード vs 論文の差分表へのリンク（あれば）
- コストモデル（多数モデル学習を伴う場合）: 総時間 ≈ N_runs × per_run_sec / 並列効率。per_run はスモークで実測した値
- 論文準拠プロトコル（再現追試の場合）: データ(規模・由来)/前処理/主要指標(定義式)/baseline/主要パラメータ、
  および照合する論文の図表番号と報告値
- 逸脱ログ（論文設定との差分）: 項目 / 論文の設定 / 本追試の設定 / 理由 / 本文での扱い
- 目的: レポートのどの主張を支えるか（章・節への参照）
- 仮説: 期待する結果と、棄却された場合の解釈
- 指標: 名称・定義・**出典（既存研究・先行評価のどれに準拠するか。再現追試なら論文の主要指標を必ず含む）**
- 比較条件: 対象（手法・実装）× パラメータ × データ。全対象に共通の条件を明記
- 公平性の担保: 同一データ・同一前処理・同一パラメータ・同一シード集合・チューニングの扱い
- 実行: スクリプト・所要時間目安・シード数
- 完了基準: 何が得られたら結果章に書けるか（再現追試なら論文の報告値と並べた表）
```

## チェックリスト

### 論文準拠（再現追試の場合・最優先）
- [ ] 再現ターゲット（exact / trend）を冒頭で宣言し、理由（RNG 非再現性・縮小規模・実装差）を添えたか
- [ ] 上流コードと論文の差分（実験条件・データ規模・試行数・乱数）を差分表に開示したか
- [ ] 論文の実験セクションを**一次情報として読み**、データ(規模・由来)・前処理・主要指標・baseline・主要パラメータを抽出したか
- [ ] 主要指標は**論文の主要指標と一致**するか（論文が予測誤差なら予測誤差で評価。独自指標は併記に留める）
- [ ] 前処理（欠損の扱い・除外・スケーリング）を論文に合わせたか。コードの example/tutorial を鵜呑みにして論文と食い違っていないか
- [ ] 照合対象の論文の報告値（図表番号）を引用し、自分の結果と並べる用意があるか（匿名データは規模一致で同定）
- [ ] 論文から逸脱する点（前処理の改善・指標変更を含む）は「逸脱ログ」に記録し、本体ではなく補助タブで扱う計画か
- [ ] 公式実装に無い比較手法・他データへの拡張は **best-effort** と位置づけ、「絶対値の厳密一致は保証しない」注記と
      論文報告値との並記を計画したか（成果・ロバストネス・別前処理は `extensions` タブで扱う）

### 指標の選定（既存研究準拠）
- [ ] 各指標に出典があるか（その分野の標準指標・先行評価のどれに準拠するか）
- [ ] 独自指標・変形を使う場合、その理由と既存指標との関係が書かれているか
- [ ] 指標が主張と対応しているか（何を測るための指標かが明確か）

### 比較の公平性
- [ ] 全対象に同一のデータ・前処理・サンプリング・主要パラメータを適用しているか
- [ ] 一方にだけ有利なチューニング（ハイパーパラメータ調整）をしていないか。する場合は全対象に同じ手続きか
- [ ] 乱数シードは複数用意し、対象間で同じシード集合を使うか（計算コストで数を変える場合は明記)
- [ ] ベースライン（最も単純な手法・素朴な比較対象）が含まれているか
- [ ] 「対象の既定値 vs 推奨値」のどちらで比較するかを宣言しているか

### 想定ユースケースを反映した評価シナリオ
- [ ] 想定する利用シーン（誰が何のために使うか）を 1 行で書いたか
- [ ] その利用シーンで実際に問われる評価になっているか
      （想定タスクで効く指標・想定する制約のもとでの挙動を測れているか）
- [ ] 失敗モード（実運用で困る壊れ方）を測る実験があるか
- [ ] 結果を実利用の判断に翻訳する一文が用意できるか（「この条件なら採用判断の材料になる」等）

### 実行可能性
- [ ] 多数モデル学習を伴う実験はコスト式 `N_runs × per_run_sec / 並列効率` を書き、per_run を
      **スモークで実測 → 総時間を宣言**してから着手する計画か（§0.5）
- [ ] 並列効率の現実性（I/O 直列化・BLAS オーバーサブスクリプション等で理論値どおりにならない）を
      想定し、縮小時の統計的代償と完走 vs 縮小の合意手順を計画したか
- [ ] 所要時間・計算資源の見積もりがあるか（重い対象はシード数を減らす等の調整を計画段階で決める）
- [ ] 既存スクリプトの拡張で済むか、新スクリプトが要るか
- [ ] 中間生成物の保存先（outputs/ experiments/ figures/）が決まっているか

### 作業 issue の起案（実行への受け渡し）

計画が固まったら、worktree で実施する**作業単位ごとに issue を起案**する
（規則は規約「作業単位の規則（worktree + issue）」）。issue 本文に含めるもの:

- [ ] 目的（計画書 `docs/plans/<ID>-...` への参照）
- [ ] 完了条件（テスト可能な形。実装を伴うなら TDD のテスト観点、実行のみなら生成物・exit code 等の機械的検証）
- [ ] 担当エージェント（通常 `implementer`）と想定 worktree／ブランチ名（issue 番号入り）
- [ ] 既存 GitHub Issue との対応（既存課題の具体化なら相互参照）

## 出力形式

```
## experiment-plan 結果
### 作成/更新した計画書
- docs/plans/<ID>-<slug>.md
### 公平性・指標に関する判断（要確認事項）
- [重要度] 論点と推奨
```
