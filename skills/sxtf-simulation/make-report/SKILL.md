# 経営層向けレポート生成手順

## 前提

- sensitivity.py / report.py が未実装の場合は、各ファイルの docstring の実装タスクを先に実装する
- 図を作る前に **dataviz スキルを読む**(チャート設計規約)

## 生成手順

```bash
uv run sxtf-sim matrix --reps 30                      # baseline 比較
# reg2027 年次ループ(runner.py の TODO 実装後): レベル別コスト曲線データ
# sensitivity.tornado(実装後): トルネード図データ
```

出力はすべて outputs/ に置く(gitignore 対象。レポート本文に転記して残す)。

## 表現規約(必ず守る)

1. **円換算を前面に**。人時・トン日は補足扱い。全国換算(scale=True)を主表示にし、
   標本ネットワーク値は脚注へ
2. **主張は2段構え**(仕様書 §0):
   - 現状比較: L0→L1→L2 の年間コスト差
   - 将来シナリオ: 規制要求の増加に対し L0/L1 のコストは比例増、L2 はほぼ平坦
3. **中心仮説を数値で**: 「L1→L2(構造化)の価値は L0→L1(単なる電子化)の◯倍」を
   必ず1文で言えるようにする
4. **レンジで語る**: 点推定を出さない。「照合工数が15〜60分のレンジで、価値は◯〜◯億円/年」
   の形式。トルネード図を根拠として添付
5. **正直さ**: キャリブレーション確度(docs/calibration-log.md)が「低」のパラメータに
   依存する結論には、その旨の注記を付ける
6. L2 の価値は必ず platform_cost 控除後(net)で提示。「プラットフォーム費用がいくらまでなら
   投資が正当化されるか」の損益分岐も添える
7. **静的計算との分解**: 件数×ホップ×工数×単価の静的ベースラインを自分で先に提示し、
   DES結果との差分を「待ち合わせ・テール・動態の効果」として分離する。
   差分が小さい項目は正直にそう書く(隠すと「複雑な掛け算」批判を受ける)
8. **閾値の逆算を主武器に**: 「照合工数が◯分を超えるなら L2 投資は正当化される」形式で提示し、
   読み手が実務感覚で検証できるようにする。点推定の押し付けをしない
9. **価値のノード別帰属表**を必ず入れる(製鉄所/商社/コイルセンター/需要家)。
   全体額より按分可能性が合意形成には効く
10. 商社の情報仲介価値の希薄化は**定量モデルに含めず**、定性章で扱う
    (新しい役割: データ品質保証・与信・在庫リスク負担)

## 検収前チェック

- 数字がレポート内で矛盾していないか(比較表とコスト曲線の初年度値が一致するか)
- OTIF が 1.0 に張り付いたままの結果を納期価値ゼロとして報告していないか
  (その場合は promised_leadtime_h の設定問題 — /verify-sim 参照)
