# パラメータ・キャリブレーション手順

現在の config/levels/*.yaml の値は仕様書 §4 の**暫定レンジ**。レポートの信頼性は
この裏付けで決まる。作業は1パラメータずつ、以下の順で行う。

## 手順

1. **docs/calibration-log.md** の該当行を確認(未記載なら行を追加)
2. 根拠を調査。優先順:
   - 公的統計・業界レポート(worldsteel、OECD Steel Outlook、日本鉄鋼連盟、経産省)
   - 学術論文(ミルシート照合工数などの実務データはほぼ無い — その場合は
     「類似業務の工数調査」からの類推であることを明記)
   - ヒアリング結果(ユーザーから提供された場合)
3. YAML の値を更新し、**行末コメントに出典を書く**(例: `# 出典: JISF 2024年報 p.XX`)
4. docs/calibration-log.md に「値、出典、確度(高/中/低)、日付」を記録
5. `/verify-sim` の手順2(matrix 再実行)で結論の向きが変わらないか確認。
   変わる場合はその旨をユーザーに報告(隠さない)

## 重要な原則

- **点推定にしない**。裏付けが弱いパラメータほどレンジ(min, mode, max)を広く取り、
  感度分析に委ねる。「不明だから適当な1点」は禁止
- 実測データが存在しないパラメータ(verify_time_min 等)は、確度「低」のまま
  レンジを維持してよい。レポート側で「感度分析済み」として扱う設計になっている
- 変更が3パラメータを超えたら、変更前後の matrix 比較を outputs/ に残す

## 検算に使える基準値(config/base.yaml と整合させる)

- 日本の粗鋼生産量: 84.0 Mt(2024年、worldsteel)
- 平均ロットサイズ 20t → 対象カバー率25%で年間約100万件の証明書、が桁の目安
