# case-study（事例再現の標準手順）

本リポジトリで事例（`case01`, `case02`, …）を再現するときの、実証済みの進め方。
プロジェクトの芯・環境ハマりどころは [`CLAUDE.md`](../../../CLAUDE.md) を前提とする。

## 原則（必ず守る）

- 位置づけは「事務局レポートを**参考に**作成した、加工の考え方のデモ」。
- **一次情報（原本 PDF）を確認せずに、事例の事実（事例番号・データ項目・加工方法）を書かない**（正確性は維持）。
- 受講者向けの本文に**出典区分ラベル（〔報告〕〔独自〕等）は付けない**。出典は各ページ末に簡潔に示す。
- デモ都合の設定（乱数 seed・分布・人工的傾向）は受講者向け本文で強調せず、開発者向けメモ（`02_*` 等）に隔離する。
- `docs/` を単一の正本にする。③の結果は②Notebookの実行出力から作る（二重実装しない）。
- 各ステップの成果物を作るたびに CI ゲート（`CLAUDE.md` 参照）をローカルで通す。

## ドキュメント構成（層① / `docs/<case>/`）

|----------|------|----------|

| `03_table_definition_before.md` | 加工前テーブル定義（ER図・実データ head サンプル） | データ概要理解 |
| `04_column_classification.md` | 情報特性の評価（識別性・機微度・結合キー性の概念とざっくり評価） | データ詳細理解 |
| `05_processing_design.md` | 加工設計（2カラム加工方針図＋評価×方針マスター表） | 加工設計 |
| `06_processing_spec.md` | 加工仕様（pandas 実装方針） | 加工仕様 |
| `07_table_definition_after.md` | 加工後テーブル定義＋前後対応表 | 結果確認 |

## UX 標準（case01 で確立。case02 も踏襲する）

スタイルは `docs/stylesheets/extra.css` に集約し、Material の CSS 変数でライト/ダーク両対応にする。描画は `mkdocs build` 後に Playwright（`/opt/node22/lib/node_modules`、`file://` で site を開く）でスクショ確認できる。

- **加工プロセスのウィザード**（`.wizard`）: 各フローページ先頭に配置。番号バッジ＋コネクタ＋完了/現在/これからの3状態、各ステップは対応ページへのリンク、現在地は `**強調**`（CSS `:has(> strong)` で判定）。ページタイトルに進行度 `— n/N`。
- **トップページ**（`index.md`）: 主CTA「① 全体概要からはじめる」＋行程ウィザード（`.wizard--overview`、現在地なし・全ステップがメニュー）＋3層×ステップ対応表。
- **05 の2カラム加工方針図**（`.proc-*`）: 左＝加工方針列／右＝「加工前 → 加工後」列。全項目を前後表示（無加工は同じ箱、削除は `***`）、方針セルを色分け。
- **05 の評価×方針マスター表**（`.master`）: `項目｜識別性｜機微度｜必要性｜必要粒度｜加工方針｜加工の根拠｜理由`。加工方針セルを色分け（`.cell-chg`/`.cell-del`/`.cell-keep`）、「加工の根拠」は規則で必須＝`.rule` バッジ（施行規則の号数）／自主的（予防的）＝`.vol` バッジ／加工なし＝`―`。規則必須の行は `tr.is-required`。
- **1枚チートシート**（`cheatsheet.md`）と**用語集**（`glossary.md`）。詳細な表は `pymdownx.details`（`??? note`）で折り畳む。文章は簡潔に。

## 手順

### 0. 一次情報の確定
- 対象レポート・事例を `WebSearch` 等で特定（本文取得はプロキシで 403 になりうる。→ ユーザーに PDF 提供を依頼）。
- 原本 PDF を確認（AES 暗号化。`uv run --with pypdf --with cryptography` で `PdfReader(...).decrypt('')` → `extract_text()`）。
- `docs/references/README.md` に書誌情報（資料名・公開主体・時期・URL・参照日・参照箇所）を登録。原本を格納する場合は出典明記。

### 1. 事例整理（層①の起点）
- `01_case_summary.md`: 事例背景・元データ項目・利用目的・必要粒度・レポート上の加工方法。制度の一行定義（仮名/匿名とは何か・両者の違い）を本文冒頭に置く。

### 2. データモデルと生成器（先に実装してスキーマを確定）
- データモデルは図表の構造を尊重して正規化（例: 顧客マスタ／履歴テーブルを結合キーで関連づけ）。
- `src/dummy_data/<case>.py`: seed・基準日固定。加工後分析が成立する人工傾向を作り込む。**読みやすい手書き生成器を基本**とする（`synthesize` パイプラインは仕様が複雑・未知のときのみ。判断は `.claude/skills/README.md`）。
- 生成 → `data/<case>/raw/*.csv` にコミット。**再生成がバイト一致**することと、狙った傾向を `groupby` で確認。
- `tests/test_<case>_dummy_data.py`: 再現性・スキーマ・PK一意・参照整合性・値域。

### 3. 設計文書 03〜08（生成スキーマに整合させる）
- 上記「ドキュメント構成」「UX 標準」に沿って作成。生成した実スキーマ・実データと必ず一致させる（doc↔data ドリフト防止）。

### 4. Notebook（層②）と結果（層③）
- `notebooks/<case>.ipynb` は `nbformat` で組み立てると壊れにくい。データは GitHub raw URL から取得。
- 段階加工 → 前後比較 → 加工後分析 → assert 確認テスト、の構成（08準拠）。
- **実行検証**: `DATA_BASE` をローカル `data/<case>/raw` に置換したコピーを `jupyter nbconvert --execute` で回し、0 エラー＋assert セル合格を確認。
- ipynb は ruff format 対象。整形後にもう一度実行検証する。Notebook の markdown セルにも出典ラベルを出さない。
- `09_results.md`: 実行済みデータの実数値（前後比較・チェック結果・加工後分析）を反映。

### 5. 仕上げ
- サイト nav（`mkdocs.yml`）に新規ページを追加（フロー＝①②③、参考＝02・08）。README の事例表・Colab バッジ・index の行程ウィザードを更新。
- 設計文書間・doc↔data↔notebook の整合をレビュー（stale/矛盾なし）。CI ゲートを全て通す。

## 匿名加工情報（case02）で変わる点

匿名加工情報は仮名加工情報より**加工の基準が厳しく、目的も逆**（元の個人へたどれる状態を残すのではなく、**再識別・連結を断つ**）。case01 の型を流用しつつ、以下を反映する。**制度の条番号・要件は必ず最新のガイドライン／施行規則で確認**すること（改正が入る）。

- **作成基準＝施行規則第34条（5基準）**を「加工の根拠」に対応づける:
  1. 特定の個人を識別できる記述等の削除
  2. 個人識別符号の削除
  3. **情報を相互に連結する符号の削除** — 仮名の「整理番号で個人単位の履歴を残す」とは逆に、**復元・連結に使える符号を残さない**
  4. **特異な記述等の削除**（外れ値・希少値。例: 極端な購入額、珍しい属性）
  5. **DB の性質を踏まえたその他の措置**（丸めの強化・トップコーディング・該当者の少ない値の削除 等。水準は**該当者の人数・データセットの大きさ・人口の多寡に応じて判断**する）
- **復元不可・識別行為の禁止**（作成者・受領者とも）。加工方法等情報の安全管理。
- **公表義務**: 作成時に匿名加工情報に含まれる情報の項目を公表／第三者提供時に項目・提供方法を公表し、匿名加工情報である旨を明示。
- **技術面**: 準識別子の組合せによる再識別リスクを、**該当者の人数・データセットの大きさに応じた一般化**と特異値処理で下げる（レポートは固定の基準値 k ではなく、該当者数・データ量に応じた可変の丸めを想定）。加工後データ内では仮IDでレコードを束ねられるが、**元の個人へはたどれない**ようにする。
- **教材上の見せ方**: 05 マスター表の「加工の根拠」は 34条の号数（規則で必須）と 5号措置（丸め等）を区別。加工後チェックは「**元の個人に戻せない（連結符号を残さない）／特異な個人が浮かない（該当者の人数を確認）**」ことを示す。データモデルにも復元符号を残さない。
- 構成は case01 に準じるが、**制度上・技術上プロセスが異なる部分は無理に同一構成へ統一しない**（spec.md 第23.7節）。

## 完了の目安

- 別の実装者が設計文書だけで Notebook を実装できる（spec §23.3）。
- Colab で上から実行して完了する。加工前後で何が失われ・維持されたか、利用目的（仮名＝個人単位分析／匿名＝再識別を断った上での利活用）が成立することが分かる。
- 加工の根拠が「規則で必須／自主的（予防的）／（匿名は 5号措置）」で読み分けられる。
