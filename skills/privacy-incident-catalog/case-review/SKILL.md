---

指定された case.json について、source に記載された公開情報との整合性を検証してください。

## 目的

case.json の記載が公開情報に照らして正確かどうかを監査する。
記事を魅力的にすることではなく、**公開情報で確認できる範囲を逸脱していないかを検証すること**が目的。

## レビューペルソナ

以下の3つのペルソナの観点すべてから検証すること。

### 1. ファクトチェッカー
- 出典URLと case.json の事実整合性を厳密に検証
- 主体・関係性・時系列が正確か
- 明記されていない因果関係を推定していないか

### 2. ビジネスレビュアー
- 一般的なビジネスユーザーが読んで誤解しないか
- 影響・被害の過剰表現がないか
- 事件の深刻度（severity）の判定が妥当か

### 3. 法規制レビュアー
- 個人情報保護法、GDPR等の法規制の記述が正確か
- 法的判断の記述が出典を超えていないか
- 罰金額、制裁内容等の事実が正確か

## 手順

### 1. case.json を読み込む

$ARGUMENTS がパスなら直接読む。事例IDなら `public/cases/{ID}/case.json` を読む。

### 2. source のURLをすべて確認する

- source にURLが複数ある場合は **必ずそのすべてを参照** する
- URLごとに、どの記述の根拠になっているかを意識して確認する
- **各URLが当該事例に直接関連する記事かを確認する**（同じメディアの別記事や、類似テーマだが別事例の記事は指摘する）
- URLがアクセス不能なら、その旨を明記する

### 3. 以下の観点で検証する

#### 3-0. スコープ適合性
- この事例がカタログの対象範囲内かを判定する
- **対象外の例**: 単なる情報漏洩（不正アクセス・標的型攻撃による流出でデータ活用の文脈がないもの）、従業員個人の不正行為（組織のデータ活用方針と無関係な個人の行為）
- 対象外と判断した場合、総合判定を「事例として不適切」とし、削除を推奨する

#### 3-1. 事実関係
- 何が起こったか（summary）が出典と一致するか
- 時系列が正確か
- 関係者・組織名が正確か

#### 3-2. 影響・被害
- impact の記述が出典で確認できるか
- 被害規模（件数、金額等）が正確か
- 推測で影響を追加していないか

#### 3-3. 原因分析
- root_cause が出典で確認できるか
- 出典にない原因を補っていないか

#### 3-4. 対応・結末
- response が出典で確認できるか
- 進行中の対応を完了したように書いていないか

#### 3-5. 教訓
- lessons_learned が事実に基づいているか
- 出典にない教訓を独自に追加していないか

#### 3-6. 深刻度
- severity の判定が影響の大きさと一致するか

#### 3-7. 可読性
- summary と impact だけ読んで事例の概要が分かるか
- summary は事件の経緯に、impact は影響・被害にそれぞれ焦点を当てているか
- ビジネスユーザーが「何が起こって、どう影響したか」を自然に読み取れるか

## 4. 法的リスク・表現チェック

#### 4-1. 価値判断表現
- 「欠如していた」→「不十分であった」「整備されていなかった」
- 「悪用」→「不適切な利用」「問題視された利用」
- 「裏切り」「怠慢」「杜撰」「悪質」→ 中立表現に置換
- root_cause と lessons_learned で特に注意

#### 4-2. 帰属表現
- 報道ベースの情報は「〜と報じられた」「〜とされている」
- 裁判確定・公式発表に基づく場合は「〜と認定された」「〜が確認された」
- 企業・当局の見解は「〜と説明した」「〜と主張した」

#### 4-3. 係争中事案
- response に「異議」「上訴」「係争中」が含まれる場合、summary/root_cause に断定的表現がないか確認
- 結論が出ていない調査・訴訟を確定事実として記述していないか

#### 4-4. 個人名・個人情報
- 被害者（再識別された個人、漏洩被害者）の実名は原則削除
- 告発者・公人（CEO等）の氏名は掲載可だが必要性を確認
- 未成年者に関する記述は年齢・性別・地域の特定を避ける

## 5. 倫理・配慮チェック

#### 5-1. センシティブデータの列挙
- HIV、性的指向、中絶、薬物使用等の具体的列挙は必要最小限に
- 「特別カテゴリーデータ」「極めてセンシティブな情報」等の総称を活用
- 同一のセンシティブ用語が summary/impact/root_cause で不必要に繰り返されていないか

#### 5-2. バランス・公平性
- response フィールドに当事企業の改善努力・反論が記載されているか
- 一方的な非難になっていないか
- 特定の国・地域名が不当に繰り返されていないか

#### 5-3. 教育的価値
- lessons_learned が建設的で再発防止に焦点を当てているか（単なる批判ではないか）
- 犯罪の模倣に利用可能な具体的手法を記述していないか

## 6. データ品質チェック

#### 6-1. 出典の量と質
- sources が1件のみの場合、追加出典の補充を推奨
- Wikipedia のみが出典の場合は警告
- source.title が10文字以下の場合は警告（正式な記事タイトルに修正）
- source_type と URL の整合性（.pdf → "pdf"）

#### 6-2. 数値・日付の検証
- occurred_at の基準（発覚日 or 行為開始日）が他事例と統一されているか
- 漏洩件数・制裁金額が出典と一致するか
- 為替換算の時期・レートが明記されているか

#### 6-3. 分類の妥当性
- severity が判定基準に合致するか
- incident_category の選択が事例内容に合致するか
- domain の選択が適切か

#### 6-4. 用語の正確性
- ドメイン固有の用語が正確に使われているか
- 顔識別（1対N照合・データベース照合）と顔認証（1対1照合・本人確認）の使い分け
- その他、技術用語・法律用語が適切か

## 最重要ルール（厳守）

1. **明記されていない因果関係は推定しない**
2. **弱い表現は弱いまま扱う** — 「疑い」「可能性」は「確定」ではない
3. **推測で補完しない** — 不足部分を常識で埋めない
4. **出典依存で評価する** — source に記載された公開情報のみで判断
5. **価値判断を含めない** — 「欠如」「悪用」「裏切り」等の表現を使わない
6. **被害者の実名を含めない** — 再識別被害者・漏洩被害者の匿名化を確認

## 出力形式

```markdown
## レビュー結果: {事例ID}

### 総合判定
正確 / 概ね正確 / 要修正 / 根拠不足が多い / 事例として不適切

### 総評
（2〜6文）

### source確認結果
| URL | 参照可否 | 主な用途 | 備考 |

### 項目別レビュー
| 項目 | case.jsonの記載 | 判定 | 根拠URL | 指摘内容 | 修正案 |

### スコープ適合性
- カタログ対象: 適切 / 対象外（→ 削除推奨）
- 対象外理由:（該当時のみ記載）

### 特に重要な懸念
- 事実関係の誤り: あり/なし
- 影響の過剰表現: あり/なし
- 原因の推測: あり/なし
- severity の妥当性: 適切/要見直し
- 可読性: 概要が伝わるか

### 法的リスク・表現
- 価値判断表現（「欠如」「悪用」等）: あり/なし
- 帰属表現の適切さ: 適切/要修正
- 係争中事案の断定: あり/なし
- 被害者個人名の記載: あり/なし

### 倫理・配慮
- センシティブデータの過度な列挙: あり/なし
- 企業の反論・改善努力の記載: あり/なし
- 教育的価値（lessons_learned）: 建設的/要改善

### 出典の関連性
- 無関係な出典の混入: あり/なし
- 該当URL:（ありの場合のみ記載）

### 用語の正確性
- 不適切な用語使用: あり/なし
- 指摘事項:（ありの場合のみ記載）
```

## 7. レビュー完了フロー

レビューの結果、総合判定が「正確」または「概ね正確」（軽微な修正のみ）の場合:

1. 指摘事項があれば case.json を修正する
2. `review_status` を `"human_reviewed"` に変更する
3. title から「【未レビュー】」を削除する
4. `npm run validate` でバリデーションを確認する

総合判定が「事例として不適切」の場合:
1. `review_status` を `"flagged"` に変更する
2. 事例の削除を推奨し、ユーザーに確認する
