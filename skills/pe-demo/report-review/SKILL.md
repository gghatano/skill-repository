# 実験レポートのレビュー（前提・結果・考察・課題）

`.claude/docs/documentation-conventions.md` の第 2 節「記述の分離原則」を前提に、
レポート系 Markdown（本体 `content/REPORT.md`・補助タブ `content/*.md`）を点検する。

## 手順

1. 対象ファイルを読み、章構成を把握する。レポート本体の標準構成:
   アブストラクト → イントロ（貢献の箇条書き）→ 関連技術・前提 → 実験設定 → 結果 → 考察 → 残課題 → 結論 → 再現補足 → 参考文献。
2. 以下のチェックリストで点検する。
3. 表記ゆれ・リンク切れ・数値転記ミスなどの軽微な問題はその場で修正する。
   主張内容の変更を伴う指摘（考察の妥当性、結論の強さ）は修正せず報告に留める。
4. **日本語本文の AI 臭を点検する**。`stop-ai-slop-jp` スキル
   （`.claude/skills/stop-ai-slop-jp/`）を併用し、主体の不在（false agency）・命題型 H2・
   両論併記・強度の均一さ・記号アーティファクトなどを直す（詳細は下のチェックリストと当該スキル）。
5. 新たな残課題を見つけたら GitHub Issue 化を提案する。

## チェックリスト

### 構成・論理の流れ・可読性（アカデミック体裁）
- [ ] **動機→定義→実験の論理**: 実験設定章は「何を知りたいか（問い）→ そのために何を測るか（指標・データ・条件）」の順で、各問いが結果章の各実験に一対一で対応づくか（問いの明示がなく手段から入っていないか）
- [ ] **実験の俯瞰**: 結果章の冒頭に、各実験（何の問い／データ／指標／期待／主結果）を一望できる前置き段落か表があるか
- [ ] **図表は本文で導入してから提示**: 各結果小節が「やったこと（本文1–2文）→ 図・表 → 事実 → 考察」の順か（見出し直後にいきなり図が来ていないか）。図表に通し番号付きキャプション（図 5.1 / 表 5.1）があり本文から参照されるか
- [ ] **箇条書きの階層**: ネストの深さ・粒度が揃っているか（一項目だけ多階層／突出して長い項目がないか。長すぎる項目は小表や独立項目へ）
- [ ] **冗長・重複**: 同一論点の三重記述や、編集ミスの重複行がないか（結果考察・§考察・結論で同じ説明を繰り返していないか）

### 前提（仕様・ドキュメント由来の記述）
- [ ] 上流ドキュメント・コード由来の記述に出典タグ [n] が付いており、参考文献章に対応があるか
- [ ] スコープ宣言（評価対象 API・対象外機能）が README とレポート本文で一致しているか
- [ ] 「ドキュメント記載の事実」と「コードからの推定」が区別されているか（推定には 🔎 考察ブロック等のマーカー）

### 結果（実測値）
- [ ] 再現ターゲット（exact / trend）が宣言され、上流コードと論文の差分が表で開示されているか
      （規約 §4「乱数再現性と再現ターゲット」）
- [ ] 結果が誇張なく書かれているか（非決定的な処理を含む場合「傾向の再現」として書き、単一 run の値を
      「完全一致を達成」等と過大表現していないか）
- [ ] 数値に実行条件（データ・サンプル数・seed・主要パラメータ・依存バージョン）が紐付いているか
- [ ] 同じ指標が複数ページに出る場合、値が一致しているか（本体 ↔ 発展・追加実験 ↔ 手法解説ページ）
- [ ] 図（figures/）と本文の数値・キャプションが一致しているか
- [ ] 単一シードの値を断定的に扱っていないか（揺れの注記、または mean±std の併記)
- [ ] **単一seed値と複数seed(mean±std)値を同一の比較・分解・梯子に混在させていないか**
      （混ぜる場合は各数値に (single-seed) / (n=k) を明示。注記があっても混在自体を避ける）
- [ ] **単一seedのアンカーを使う箇所で、同一条件の複数seed測定が別途存在しないか**
      （存在するなら平均で置換。少なくとも、そのアンカーが複数seed分布の最良/最悪 draw で
      ないかを照合し、最良 draw を分解・断定の土台にしていないか）

### 考察
- [ ] 各考察が、どの結果（実験 A/B/... や表番号）を根拠にしているか明示されているか
- [ ] 結果が支持する範囲を超えた一般化をしていないか（「本実験条件では」等の限定があるか）
- [ ] 手法・条件間の差を論じる際、ばらつき（std）に対して差が有意かに触れているか
- [ ] **差の断定に機械的な線引きを適用しているか**: 主張する差が計測済み std と同オーダー
      （差 < std）、または分布が重なる（劣位側の max ≥ 基準の mean）とき、「悪化／優劣」と
      断定せず判定保留にしているか。分解（gap を A 分＋B 分に切り分ける等）の各成分が std を
      上回っているか
- [ ] **方向性の結論（A が勝つ／悪化する）が、単一seedしか無い条件に依存していないか**
      （その条件に std が無いこと自体を根拠不足として扱う。「単一seedは過信しない」と
      書きながら単一seedの点差を決め手にする等の内部矛盾がないか）
- [ ] 構成要素のアルゴリズム固有の深掘り（手法 A が B を上回る理由・構造的限界・計算コスト等）が本体に残らず個別ページに分離され、本体は検証対象主軸の要約＋リンクになっているか

### 課題
- [ ] 残課題章の項目が GitHub Issues と対応しているか（漏れ・重複）
- [ ] 対応済みの課題に ✅ と残作業が追記されているか
- [ ] 適用範囲の限界（単一データセット・特定バージョン等）が結論・アブストラクトに反映されているか

### AI 臭の除去（日本語としての自然さ）
詳細・修正例は `stop-ai-slop-jp` スキル（`references/structures.md`・`phrases.md`・`examples.md`）に委ねる。
レポートでは特に次を見る（学術体裁を保ちつつ）:
- [ ] **主体の不在（false agency）**: モノが人間の動詞を取っていないか（「データが示す」「結果が物語る」→ 誰が何をしたか・何が観測されたか）
- [ ] **命題型 H2**: 見出しが「○○は△△だ」型の主張になっていないか（名詞句で立てる）
- [ ] **両論併記の放棄回避**: 「○○もあり、△△もある」で判断を回避していないか（採用した設定・解釈を引き受ける。§2 の事実/考察分離と接続）
- [ ] **強度・リズムの均一さ**: 全段落が同じ長さ・同じ強度・同じ文末で揃っていないか（考察は「〜と考えられる」等、事実章と語調を変える）
- [ ] **偏愛語・必殺技造語・横文字メタファー**: 「本質」「解像度」「手触り」「思考の OS」等を撒いていないか
- [ ] **記号アーティファクト**: 全角ダッシュ `──`・装飾絵文字の等間隔撒布・`**` 残骸・コロン後の半角スペースが残っていないか

> アカデミック体裁との両立: 出典タグ [n]・図表番号・用語定義は残す。AI 臭除去は「文体の人間化」であって、事実の裏取りや厳密さを崩さない。

```
### Issue 化候補
- ...
```
