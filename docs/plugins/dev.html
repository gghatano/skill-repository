開発ワークフロー（Git / Issue / PR）

GitHub Issue を作業単位に、branch / worktree / 実装 / 検証 / レビュー / PR を一貫管理する開発フローのスキル群。どのリポジトリ種別にも適用できる汎用ワークフロー。

/plugin install dev@gghatano-skills

含まれるスキル（9）

  • code-review実装差分を docs/review-checklist.md に沿ってレビューする手順。実装完了前に必ず使う。
  • issue-driven-developmentGitHub Issue を作業単位として branch/worktree/PR/検証記録を一貫管理する。実装作業を始めるときに必ず使う。
  • issue-triage仕様や発見事項から Issue 候補を抽出し、適切な粒度・依存・ラベルに整理する手順。Issue を作る前の計画時に使う。
  • pr-from-issueIssue に紐づく PR を作成し、Issue へ進捗・検証結果をコメントして相互リンクする手順。作業完了時に使う。
  • pr-prepPR 作成前の最終確認手順。コミット／PR を出す直前に必ず使う。
  • refactor-safely振る舞いを変えずに内部構造だけ改善する手順。リファクタリングを実装タスクと分けて行うとき使う。
  • spec-implementationdocs/SPEC.md に基づき、実装タスクを小さく切り出して実装・検証する手順。仕様から1単位を実装するとき必ず使う。
  • test-fix-loopテスト・lint・型チェックの失敗を分析し、修正し、再実行して緑にする手順。検証が落ちたとき使う。
  • worktree-task-runnerIssue ごとに専用 branch と git worktree を作成し、分離した作業ディレクトリで実装する手順。Issue 着手時に使う。

同梱物（規約・エージェント）

docs/dev-workflow-conventions.md

導入後の調整

docs/dev-workflow-conventions.md の §1 検証コマンド・§2 ブランチ方針・§3 参照ドキュメントを、対象リポジトリの実際のコマンド・パスに合わせて調整してください。code-reviewer / security-reviewer 等のサブエージェントを使う場合は、そのプロジェクトの .claude/agents/ に用意してください（無ければ通常のレビューで代替されます）。

導入手順

  1. マーケットプレイスを追加（初回のみ）/plugin marketplace add gghatano/skill-repository
  2. このプラグインを導入 /plugin install dev@gghatano-skills
  3. スキルを名前空間付きで呼び出す（例 /dev:code-review）
