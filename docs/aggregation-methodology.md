# 取りまとめの方法論 — Input / Process / Output と検証（Validation）

このドキュメントは、本リポジトリが行っている「複数リポジトリに散らばった資産（スキル）を集めて、
検証済みのカタログと配布物にまとめ上げる」取りまとめの**手順と考え方**を、
別の環境・別のドメインでも再現できる形で整理したものです。

対象読者は、次のいずれかを行いたい人です。

- 同じ仕組みを **別のアカウント / 別のリポジトリ群** に対して回したい
- スキル以外の資産（テンプレート、データセット、事例、設定ファイル…）を **同じ考え方で取りまとめたい**
- 既存パイプライン（`scripts/sync_skills.py`）の **設計意図** を理解して安全に改造したい

実装の詳細は `scripts/sync_skills.py`（約 1,000 行、1 ファイル）と `tests/test_sync_skills.py` に集約されています。
本書はその上位にある**設計思想**を言語化します。

---

## 0. 全体像 — 取りまとめを「3工程 + 検証」で捉える

取りまとめは、次の 3 工程として捉えると再現しやすくなります。各工程の**境界に検証ゲートを置く**のが要点です。

```
┌────────────┐   validate   ┌────────────┐   validate   ┌────────────┐
│   INPUT    │ ───────────▶ │  PROCESS   │ ───────────▶ │   OUTPUT   │
│  収集・取込  │   入力検証     │ 正規化・分類  │   工程内検証    │  生成・配置   │
└────────────┘              └────────────┘              └────────────┘
     │                            │                            │
  出典から集める                正規化して意味づけする            決定論的に書き出す
  （何を・どこから）             （どう揃えるか）                 （壊さず・全か無か）
```

- **INPUT（収集）**: 出典（source of truth）から取込対象を発見し、取込む。ここで「何を対象とするか」の契約を固定する。
- **PROCESS（変換）**: 揃っていない入力を**正規化**し、分類・移植性判定などの**意味づけ（enrichment）**を付与する。
- **OUTPUT（生成）**: 人間可読なカタログと機械可読なマニフェスト、そして配布物を**決定論的に**書き出す。

検証（Validation）は独立した第 4 の工程ではなく、**各工程の境界に埋め込まれた関門**です。
「途中で失敗したら、中途半端な結果を残さず、まるごと拒否する（fail-closed / all-or-nothing）」を貫くのが、この取りまとめの背骨です。

## 1. INPUT — 収集と入力検証

### 1.1 何を入力とするか（取込の契約）

まず「取り込む対象」を厳密に定義します。本リポジトリでは次を契約としています。

| 項目 | 本リポジトリでの定義 | 別環境で変える箇所 |
| --- | --- | --- |
| 出典（source） | GitHub アカウント `gghatano` の全リポジトリ | `--owner`、または列挙元そのもの |
| 取込単位 | `.claude/skills/**/SKILL.md` の存在するディレクトリ | パス規約（`SKILL_PATH_RE`） |
| 同梱物 | `SKILL.md` と同ディレクトリ配下の全ファイル（`scripts/` `references/` `assets/` 等） | 「単位ディレクトリ丸ごと」の粒度 |
| 対象ブランチ | 各リポジトリのデフォルトブランチ | `defaultBranchRef` |
| 除外 | アーカイブ済みリポジトリ、デフォルトブランチ無し | 除外条件 |

**考え方**: 取込単位は「意味のあるまとまり」に合わせる。スキルは 1 ファイルではなく、
`SKILL.md` を起点とした**ディレクトリ丸ごと**が 1 単位。これにより「本体だけコピーして付属物が欠ける」事故を構造的に防ぎます。

### 1.2 入力検証ゲート（`discover_skills`）

収集の段階で、次を**満たさなければ即座に中断**します（`SyncError` を送出）。

| 検証 | 目的 | 実装箇所の考え方 |
| --- | --- | --- |
| Git ツリーが truncated でない | 巨大リポジトリで一覧が途中で切れると**取りこぼしに気づけない** → 部分同期を拒否 | `if tree_data.get("truncated"): raise` |
| ツリーが存在し list である | API 応答の破損検知 | `if not isinstance(tree, list): raise` |
| 取込先（destination）の衝突なし | 同一リポジトリ内で同名スキルが 2 つ以上あると**上書き事故** | `destination in destinations` で検出 |
| 指定リポジトリが実在する | `--repo owner/name` の打ち間違いを**黙って空振りさせない** | フィルタと既知集合の差分を報告 |

**設計原則（INPUT 側）**:

- **部分成功を成功と見なさない**。「一部だけ取れた」は「壊れた入力」として扱う。取りまとめの信頼性は「全件を漏れなく見た」という前提に依存するため。
- **順序を固定する**。リポジトリもスキルも `sorted(...)` で正規化し、実行のたびに同じ順序で並ぶようにする（後段の決定論の土台）。

## 2. PROCESS — 正規化・意味づけと工程内検証

入力は「揃っていない」ことを前提にします。PROCESS の役目は **(a) 形を揃える（正規化）** と
**(b) 意味を足す（enrichment）** の 2 つです。

### 2.1 正規化（normalize）— 欠損に強くする

| 対象 | 揃え方 | フォールバック |
| --- | --- | --- |
| フロントマター | `---` で囲まれた YAML 風メタを緩くパース（`\|` 折り畳み・引用符除去に対応） | 無ければ空 dict |
| スキル名 `name` | フロントマターの `name` | 無ければ**ディレクトリ名**を採用 |
| 概要 `description` | フロントマターの `description` | 無ければ**本文の最初の段落**を要約、それも無ければ `"説明なし"` |
| 空白 | `" ".join(text.split())` で連続空白・改行を 1 つに畳む | — |

**考え方**: 出典側の書き方を強制しない。**「理想的に書かれていなくても取りまとめられる」**ように、
各フィールドに段階的フォールバックを用意する。これが「別環境でも、既存の資産をそのまま流用できる」再現性の源になります。

### 2.2 意味づけ（enrichment）— 分類・移植性・ツール検出

正規化した各エントリに、機械的に導ける**メタ情報**を付与します。

1. **ツール検出（`detect_tools`）**: 本文から `Python` / `uv` / `GitHub CLI` などの依存を拾う。
2. **プロジェクトパス検出（`detect_project_paths`）**: 本文が参照する具体パスを抽出（移植性判定の材料）。
3. **移植性分類（`classify_portability`）**: 検出パスを 4 バケットに振り分け、3 段階の tier を導く。

#### 移植性（portability）の判定ロジック — 取りまとめの中核的な「意味づけ」

各スキルが参照するパスを、性質で 4 つに分けます。

| バケット | 意味 | 例 |
| --- | --- | --- |
| `bundled` | スキル自身が同梱するファイル/ディレクトリ（一緒に移動する） | `scripts/gen.py`, `references/` |
| `parameterized` | プレースホルダ/グロブで**既に汎用化**されている | `$ARGUMENTS/input/`, `docs/report_{{NN}}.md`, `input/<table>.csv` |
| `companions` | `.claude/` `.agents/` `.github/` 配下の**兄弟ファイル**（一緒にコピーが必要） | `.claude/agents/reviewer.md` |
| `external` | **特定プロジェクト固有の具体パス**（移植先で読み替えが必要） | `source/raw_data/`, `docs/review.md` |

そこから tier を決めます（**より重い問題を優先**する）。

- `external` が 1 つでもあれば → **要調整（rework）**
- そうでなく `companions` があれば → **同梱前提（companions）**
- どちらも無ければ → **そのまま（portable）**

**考え方**: 「移植のしやすさ」を人手の主観ではなく、**パスの性質から機械的に導出**する。
これにより、スキルが増えても判定が一貫し、`docs/skill-porting-guide.md` として自動でチェックリスト化できます。
別ドメインへ応用するときは、この「バケット定義」を自分の資産の**外部依存の形**に合わせて置き換えます。

### 2.3 分類（categorize）— 名前ベースのタクソノミー

`catalog/categories.json` が「スキル**名** → 目的カテゴリ」の対応表です。
機械的な収集（自動）と、**意味的な分類（人が定義）**を分離しているのがポイントです。

- カテゴリ定義は人が管理（`id` / `title` / `summary` / `skills[]`）。
- 収集時に名前で突き合わせて `category` を付与。未定義は `未分類` に集約し、**警告を出す**（握りつぶさない）。

### 2.4 工程内検証ゲート

| 検証 | 目的 |
| --- | --- |
| categories.json が妥当な JSON | 壊れた分類定義で全件が `未分類` に落ちるのを防ぐ |
| `id` の重複なし | カテゴリ定義の一意性 |
| 1 スキルが 2 カテゴリに属さない | 分類の排他性（`skill 'X' is assigned to both ...`） |
| 未分類スキルの警告 | 取りこぼしを**可視化**（黙って落とさない） |

**設計原則（PROCESS 側）**:

- **自動導出できるものは自動化、意味づけが要るものだけ人手**。分類は人、移植性判定は機械、という役割分担。
- **警告は出すが止めない / エラーは止める**の線引きを明確に。未分類は警告（運用で拾う）、分類定義の破損は中断（信頼性が崩れる）。

## 3. OUTPUT — 生成物と出力検証

### 3.1 何を出力するか

1 つの入力集合から、用途別に複数の生成物を作ります。**単一の真実源（manifest）から派生物を導く**構造です。

| 生成物 | 種別 | 役割 |
| --- | --- | --- |
| `skills/<repo>/<skill>/` | ファイル群 | 同期したスキル本体（横断カタログの素材） |
| `catalog/manifest.json` | 機械可読 | **単一の真実源**。名前・出典・SHA・移植性・分類を含む |
| `docs/skill-catalog.md` | 人間可読 | 出典リポジトリ順の一覧 |
| `docs/skill-purpose-catalog.md` | 人間可読 | 目的カテゴリ別の一覧 |
| `docs/skill-porting-guide.md` | 人間可読 | 移植チェックリスト（tier 別） |
| `docs/skill-duplication.md` | 人間可読 | 複数リポジトリ重複の検出（集約候補） |
| `docs/index.html` | 配布 | GitHub Pages 一覧（テンプレートにデータ差し込み） |
| `docs/skills/<name>.html` | 配布 | 各**スキルの詳細ページ**（できること・使いどころ・入出力）。一覧のスキル名からリンク |
| `docs/plugins/<name>.html` | 配布 | 各**プラグインの詳細ページ**（含まれるスキル・同梱物・導入後の調整・導入手順）。一覧のプラグインカードからリンク |
| `plugins/<name>/` | 配布 | 目的別プラグイン（`/plugin` で導入） |

**考え方**: **manifest を真実源**にし、`--from-manifest` で GitHub を叩かずに全派生物を再生成できるようにする。
これにより「表示の作り直し」と「収集のやり直し」を分離でき、オフラインでも整合を保てます。

#### 一覧ページと詳細ページ — 生成物を「入口」と「深掘り」に分ける

配布物の HTML は 1 枚ではなく、**一覧（index）→ 詳細（detail）**の 2 層で構成します。

- **一覧ページ**（`docs/index.html`）: 全スキル・全プラグインの入口。カード/名前から詳細ページへリンクする。
- **スキル詳細ページ**（`docs/skills/<name>.html`）: スキル 1 件ごとに 1 枚。`web/skill-detail.template.html` に
  **人手で書いた要約** `catalog/skill-details.json`（`tagline` / `canDo[]` / `whenToUse` / `io`）を流し込む。
- **プラグイン詳細ページ**（`docs/plugins/<name>.html`）: プラグイン 1 件ごとに 1 枚。`web/plugin-detail.template.html` から
  生成する。**先頭に「何を入力して・何をして・何を得るか」（入力→処理→出力）の要約**を置き、含まれるスキル一覧・
  同梱物（`agents/` `docs/`）・**導入後の調整**をその後に続ける。入力/処理/出力の要約は `catalog/plugin-details.json`
  （`input` / `process` / `output`）から、調整メモは `plugins/<name>/README.md` の「導入後の調整」節を
  `extract_readme_section` で引用する。

**考え方**: 詳細ページの主役は「含まれるスキルの一覧」ではなく、**「何を入力すると何が得られるか」という価値の流れ**。
利用者がまず知りたいのは中身の部品ではなく、入力→出力の対応だからです。スキル一覧は「その流れをどう実現するか」の
補足として下に置きます。これはこのリポジトリ全体を貫く input / process / output の視点を、配布ページにも適用したものです。

**考え方**: 一覧は機械的に導ける事実（名前・概要・出典）で作り、詳細は**人手の説明を上乗せ**する。
これは 2.3 の「収集は機械 / 意味づけは人手」を**出力側にも一貫させた**もので、`skill-details.json`・`plugin-details.json` と
README の「導入後の調整」節が、`categories.json` と並ぶ**人手管理の意味づけソース**になります。
入力/処理/出力・同梱物・調整メモが無いプラグインでは、該当セクションを**省く**（空欄を晒さない）。

### 3.2 出力検証ゲート — 「壊さない」ための順序と原子性

出力工程は、**「途中で失敗しても既存の成果物を壊さない」**ことに全力を注ぎます。

| 技法 | 何をするか | なぜ |
| --- | --- | --- |
| **render-before-mutate** | `skills/` を差し替える**前に**全ドキュメント（一覧・詳細ページ・マニフェスト含む）を文字列として生成しきる | テンプレート不備等の失敗を、破壊的操作の前に検知する |
| **一時ディレクトリにステージング** | 新しい `skills/` は temp に組み立ててから入れ替える | 組み立て中の中間状態を本番に晒さない |
| **バックアップ付き入れ替え**（`replace_directory`） | `X` → `X.previous` に退避 → 新 `X` を配置 → 失敗時は**ロールバック** | 差し替えの原子性と復旧可能性 |
| **原子的書き込み**（`atomic_write`） | 一時ファイルに書いて `os.replace` で置換 | 書き込み途中のファイルを読ませない |
| **全件取得成功が前提** | 収集が全件成功してから生成物を置き換える | 部分同期で不完全なカタログを残さない |

### 3.3 決定論（determinism）— 差分をノイズにしない

- エントリは常に `(repository, name)` で**ソート**。
- マニフェストのキー順・整形も固定。
- 同じ入力なら**同じ出力**（テスト `test_manifest_and_catalog_are_deterministic` が保証）。

**考え方**: 生成物を Git 管理する以上、**実行ごとに無関係な差分が出ない**ことが運用コストを決める。
決定論はレビュー可能性そのものです。

## 4. 別環境で同じことをするための移植レシピ

「同じ取りまとめを違う環境でやる」ための手順です。**変える箇所（パラメータ）**と**変えない箇所（不変条件）**を分けて考えます。

### 4.1 変える箇所（環境ごとのパラメータ）

| パラメータ | 本リポジトリの値 | 変更方法 |
| --- | --- | --- |
| 出典アカウント | `gghatano` | `--owner NAME`（`build_parser` の既定値、または CLI 引数） |
| 取込対象のパス規約 | `.claude/skills/**/SKILL.md` | `SKILL_PATH_RE` を対象資産のパスに合わせて書き換え |
| 取込単位ファイル | `SKILL.md` | 「単位ディレクトリの目印ファイル」を差し替え |
| 目的タクソノミー | `catalog/categories.json` | 自分のドメインのカテゴリで定義し直す |
| 移植性バケット | `bundled/parameterized/companions/external` | 自分の資産の「外部依存の形」に合わせて正規表現・接頭辞を調整 |
| 詳細ページの人手要約 | `catalog/skill-details.json`（スキル）、`catalog/plugin-details.json`（プラグインの入力→処理→出力） | 各項目の説明・価値の流れを自分のドメインで用意 |
| 配布物 | 4 プラグイン + 一覧/詳細ページ | 不要なら `plugins/` や詳細ページ生成（`--skill-detail-template` / `--plugin-detail-template`）を外す |
| 出力先 | `skills/` `docs/` `docs/skills/` `docs/plugins/` `catalog/` | `build_parser` の各パス引数 |

### 4.2 変えない箇所（不変条件 = 再現性の核）

別ドメイン（例: 事例カタログ、データセット目録、設定ファイル集）に応用しても、次は**必ず引き継ぐ**べき原則です。

1. **Fail-closed**: 部分成功を成功と見なさない。取りこぼしの可能性があれば中断する。
2. **All-or-nothing の入れ替え**: 生成しきってから、原子的に、バックアップ付きで差し替える。
3. **単一の真実源**: 機械可読な manifest を中心に置き、人間可読な派生物はそこから導く。
4. **決定論**: ソートと整形を固定し、同じ入力から同じ出力を出す。
5. **正規化のフォールバック**: 出典に完全さを要求しない。欠損に段階的に耐える。
6. **自動 vs 人手の分離**: 機械的に導けるメタは自動、意味づけ（分類）だけ人が定義する。
7. **警告と中断の線引き**: 取りこぼしは可視化（警告）、信頼性が崩れる破損は中断（エラー）。
8. **テストで不変条件を固定**: 決定論・検証ゲート・移植性判定を回帰テストで守る。

### 4.3 最短の移植手順（チェックリスト）

- [ ] 取込対象の**契約**を書き出す（出典 / 取込単位 / 同梱物 / 除外条件）。
- [ ] `SKILL_PATH_RE` 相当の**パス規約**を自分の資産に合わせる。
- [ ] **入力検証ゲート**を移植する（truncated 拒否・衝突検出・存在確認）。
- [ ] **正規化**のフィールドとフォールバックを決める（名前・概要・タグ等）。
- [ ] **意味づけ**の自動導出（依存・移植性など）を、自分のドメインの関心事に置き換える。
- [ ] **タクソノミー**（`categories.json` 相当）を人手で定義し、名前で突き合わせる。
- [ ] **真実源となる manifest** のスキーマを決め、派生ドキュメントを render 関数として書く。
- [ ] 表示は**一覧→詳細の 2 層**で設計する。詳細ページの人手要約（`skill-details.json` 相当）は任意だが、あると価値が伝わりやすい。
- [ ] **出力の原子性**（render-before-mutate / staging / backup 入れ替え / atomic write）を必ず入れる。詳細ページも同じ render セットに含める。
- [ ] **決定論テスト**と**検証ゲートのテスト**を用意する。
- [ ] 定期実行（CI）で**同期 PR 自動作成**まで繋ぐと運用が回る（`.github/workflows/sync-skills.yml` 参照）。

## 5. 実装との対応表（コードの読み解きガイド）

本書の各概念が `scripts/sync_skills.py` のどこに対応するかの索引です。

| 工程 | 概念 | 主な関数 |
| --- | --- | --- |
| INPUT | 出典の列挙・取込単位の発見・入力検証 | `discover_skills`, `GhClient.list_repositories` / `get_tree` |

## 6. まとめ — 取りまとめを支える一文

> **「全件を漏れなく見た」という前提のもとで、揃っていない入力を正規化して意味づけし、
> 壊さず・全か無かで、同じ入力からは同じ出力を、単一の真実源から書き出す。**

この一文に含まれる 4 つの約束 — **網羅（fail-closed）／正規化（欠損耐性）／原子性（壊さない）／決定論（同一性）** — を
自分のドメインの資産に当てはめれば、環境が変わっても同じ品質の取りまとめが再現できます。
