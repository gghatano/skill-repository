# 実験の実行と結果の記録（再現性ファースト）

`experiment-plan` の計画書（**実験計画**。規約 `.claude/docs/documentation-conventions.md`
§1「パスの読み替え」。既定は `docs/plans/<ID>-<slug>.md`）に基づいて実行する。
計画書がない実験は、先に `experiment-plan` で計画を作る（場当たり実行をしない）。
以下に現れるスクリプト・生成物・依存固定のパスは同 §1 の役割に対する既定値であり、
別プロジェクトでは読み替える。

本プロセスは手を動かす作業が中心のため、サブエージェントに委譲する場合は
`implementer`（下位ティア、`.claude/agents/implementer.md`）を使う（同梱エージェントが
無い環境では通常のサブエージェントで代替してよい）。
ただし結果の解釈・レポートの考察執筆は implementer に任せず、本スキルの呼び出し元
（または `planner`）が行う。

作業単位の規則（規約参照）:
- **1 worktree = 1 issue = 1 サブエージェント**。対応 issue のない実行をしない
  （issue は計画時に `experiment-plan` が起案している前提。なければ差し戻す）。
- 並行実行は worktree で隔離する（Agent ツールの `isolation: "worktree"` か、issue 番号入り worktree）。
- スクリプトの実装・修正を伴う場合は **TDD**: 完了条件を `tests/`（pytest）の失敗するテストとして
  先に書き、通るように実装する。

## 実行前チェック（環境の再現性）

- [ ] 環境はプロジェクトの依存固定（**uv** の `pyproject.toml`+`uv.lock`、または `requirements.txt` 等。
      規約 §1・§4 に従う）由来か（素の pip install をしない）
- [ ] 依存を追加した場合、ロックファイル（`uv.lock` / `requirements.txt` 等）を更新したか（推移的依存まで固定、git 依存はハッシュ固定）
- [ ] 上流コード・外部ツールへの修正がある場合、所定の手順で適用された状態か
- [ ] 実行環境（OS・Python・主要パッケージのバージョン）を記録する仕組みがあるか
      （**実行メタ**。既定 `outputs/run_meta.json` 相当に保存）

## 実行の規約

1. **スクリプトの配置**: **実行スクリプト**（規約 §1。既定 `scripts/`。ビルドは各プロジェクトの
   ビルダー）配下に置き、実行順が分かる命名（番号プレフィックス等）にする。新実験は既存スクリプトへの追加か新設で対応する。
2. **パラメータ**: シード・サンプル数・主要パラメータ等はスクリプト冒頭の定数か引数に集約する
   （本文・図のキャプションから参照できるように）。計画書の条件と一致させる。
3. **シード**: 計画書のシード集合をそのまま使う。対象間で同一集合。
   途中で増減した場合は計画書に追記する。
4. **長時間実行**: バックグラウンド実行とし、完了待ちでセッションをブロックしない。
   メモリ・キャッシュ起因の揺れに注意し、対象間でプロセスを分けるか明示的にクリアする。
5. **生成物**: 指標は JSON（`outputs/` `experiments/`）、図は PNG（`figures/`）に保存。
   生成物はすべて再実行で再生成可能に保つ（手作業の加工を挟まない）。
6. **失敗の記録**: 実行中に遭遇した問題（クラッシュ・数値破綻・回避策）は、その場で
   **エンジニアリングノート**（規約 §1。既定 `content/engineering.md` 相当）の「問題 / 原因 / 対応」
   テーブルに追記する。後回しにすると失われる。

## 進捗・可観測性（長時間実験）

wall-clock で概ね数分を超えうる処理は、無人実行でも「動いているか」「いつ終わるか」が分かるようにする
（進捗が見えれば、ハング・ストール・並列オーバーサブスクリプションを ETA の異常から早期に切り分けられる）。

1. **逐次進捗**: `done/total・elapsed・eta・（あれば）retries/失敗数` を一定間隔（例: 全体の 5% ごと）で
   標準出力に出す。`print(..., flush=True)` で**バッファをフラッシュ**（バックグラウンド実行でも追えるように）。
2. **事前スモーク → 外挿**: 本実行前に小規模（例 N を 1/50）で 1 単位の所要時間を測り、総時間を**宣言**してから
   本実行する（計画段階のコストモデルは `experiment-plan` §0.5。ここはその実行時の裏取り）。
3. **冒頭サマリ**: 開始時に主要パラメータ（規模・並列数・主要設定・出力先）を 1 行で出す。
4. **完了時メタ**: 所要時間・スループット（per-unit 秒）・失敗件数・依存バージョン・乱数の再現性注記を
   `outputs/*_meta.json` に保存する（再現性レビューの根拠になる）。
5. **並列の健全性**: worker 数・スレッド固定（下記）を明示し、ETA が理論値から大きく乖離したら
   **オーバーサブスクリプションを疑う**。
6. **バックグラウンド + 監視**: フォアグラウンドで張り付かず、ログを tail して監視する。

最小コード例（逐次進捗 + flush）:

```python
import time, sys
start = time.time()
for i, unit in enumerate(units, 1):
    run_one(unit)
    if i % max(1, total // 20) == 0 or i == total:   # 5% ごと
        elapsed = time.time() - start
        eta = elapsed / i * (total - i)
        print(f"[{i}/{total}] elapsed={elapsed:.0f}s eta={eta:.0f}s retries={retries}",
              flush=True)
```

## 並列・堅牢性・vendoring

並列 worker や脆い処理を含む実行では次を定石にする（出所: 多数モデル学習の無人実行で踏んだ落とし穴）。

1. **スレッド固定（並列時は必須）**: `OMP/OPENBLAS/MKL/NUMEXPR/VECLIB_*_NUM_THREADS=1` を
   **numpy import 前**に設定（spawn された子プロセスも先頭で再適用）。worker × BLAS スレッドでコアを奪い合う
   オーバーサブスクリプションを防ぐ。
2. **spawn ガード**: multiprocessing（Windows の spawn）では `if __name__ == "__main__":` ガード必須。
   `maxtasksperchild=1` は worker 毎に重い再 import を招くため適度に大きく取る。
3. **脆い処理のリトライ＋失敗の記録**: 稀な失敗（例: 数値的に厳しい入力での `ValueError`）は
   最大 N 回リトライし、恒常失敗は NaN 行として記録して下流で除外する（黙って落とさない）。
4. **絶対パス & UTF-8 明示**: cwd 非依存の絶対パス、`PYTHONUTF8=1` ＋ `encoding="utf-8"`（cp932 回避）。
5. **上流コードの最小差分 vendoring**: 必要なコードが upstream の非 main ブランチ／experimental フォルダにある等の場合、
   `scripts/<pkg>/` に**機械的変更（相対 import 化）のみ**で取り込み、`__init__`/README に
   **出所 URL・対象ブランチ/コミット・差分方針（機械的変更のみ）**を明記する（再現性の原則と接続）。

## 結果の整理（レポートへの反映）

結果は記述の分離原則（規約 `.claude/docs/documentation-conventions.md` 第 2 節。規約が無い
プロジェクトでは同等の文書規約）に沿って書く:

1. 数値表 + 図 + **実行条件**（データ・サンプル数・シード数・主要パラメータ・依存固定への参照）をセットで書く。
2. 「所見」は結果の直後に置き、考察（推定）であることが分かる書き方にする。
   単一シードなら断定せず、複数シードなら mean±std で書く。
3. 計画書の仮説と突き合わせ、支持/棄却/判断不能を明記する。判断不能なら不足している試行を残課題へ。
4. レポート骨子の該当 TODO を消し込み、`report-skeleton` の stage を進める。
5. 計画書に実施記録（実行日・コミット・所要時間・結果の要約 1 行）を追記して閉じる。
6. 仕上げに `report-review`（数値整合）→ 必要に応じて `publish-check`（公開反映）を実行する。

## 出力形式

```
## experiment-run 結果
### 実行: 実験<ID>（計画書: docs/plans/...）
- 環境: （uv / uv.lock のコミット / 実行時間）
- コマンドと exit code
### 結果サマリ（仮説の支持/棄却/判断不能）
### レポート反映: 更新したファイルと消し込んだ TODO
### engineering.md への追記（あれば）
```
